## このプロジェクトにおける AI 活用の位置づけ

yatamux は Claude Code（Anthropic の AI コーディングエージェント）を積極的に使って開発したプロジェクトだ。
単に「コードを生成してもらう」ではなく、設計レビュー・バグ調査・リファクタリング計画といった
**思考のパートナー**として使うことを意識した。

## 開発フロー

### 1. 要件をタスクファイルに書き出す

`task.md` に実装したい機能・既知のバグを箇条書きで積む。
Claude Code はこのファイルを読んで現在の状態を把握するため、
曖昧さを排除して「前提 / 操作 / 期待結果」の形式で記述する。

```
### B-2: Ctrl+C がエージェントに届かない
- 前提: ペイン内で Claude などの AI エージェントを起動している
- 操作: Ctrl+C を押す
- 期待結果: プロセスが終了する
- 根本原因: write_all 後の flush() 漏れにより 0x03 がバッファに滞留
```

### 2. 調査フェーズ：コードを読んで原因を仮説立てる

「なぜ Ctrl+C が届かないのか」を Claude Code に問いかけると、
`pty.rs` の `write()` 実装を読み、`flush()` がないことを指摘してくれた。

人間が「全ソースを読んで怪しい場所を探す」のに比べて、
関係するファイルを横断的に検索して証拠を集めてくれる点が特に助かった。

### 3. TDD：テストを先に書く

CLAUDE.md に `テストを先に書く（Red → Green → Refactor）` と明記している。
この規約を守ることで、Claude Code が「動くコード」ではなく
「正しいコード」を書こうとするプレッシャーを与えられる。

```rust
// テスト先行の例（Pane::spawn_with_handle のモック化）
let (pty_tx, pty_rx) = mpsc::channel(16);
let mock_pty = MockPty::new(pty_tx);
let pane = Pane::spawn_with_handle(id, size, config, out_tx, notif_tx, mock_pty, pty_rx);

// ConPTY を起動せずにサーバー層を検証
pty_input_tx.send(b"\x1b[31mhello\x1b[m".to_vec()).await.unwrap();
// Grid が更新されるのを待つ...
```

### 4. リファクタリングは「なぜやるか」をセットで依頼する

コードの改善を依頼するとき、動機を一緒に伝えると出力の質が上がる。

> 「`portable_pty::Child` を直接 `Pane` に持つと外部クレートの破壊的変更に
> 脆弱になる。`PtyHandle` トレイトを作ってカプセル化したい。
> 既存テストは変更しないこと」

動機が明確であれば、Claude Code は「最小限の変更」で実装する傾向がある。
動機なしで「リファクタリングして」と頼むと、不要な変更を盛り込んでくることがある。

### 5. CLAUDE.md を育てる

プロジェクト固有の知識（アーキテクチャ・落とし穴・コーディング規約）を
`CLAUDE.md` に蓄積していくことが、会話をまたいだ品質維持の鍵だった。

実際に記録した内容の例:

- **T-01**: `tokio::sync::Mutex` を `handle_client_message` 内で使うとデッドロック
  → `std::sync::Mutex` を使う
- **T-02**: `#[tokio::test]` には必ず 120s タイムアウトを付ける
- **T-03**: `Drop` で `ProcessKiller::kill()` を呼ぶ（孤児プロセス防止）

これらは一度ハマって解決したパターンで、CLAUDE.md に書いておくことで
次の会話でも同じ間違いを繰り返さずに済んだ。

## 実際の分業イメージ

| タスク | 担当 |
|--------|------|
| 何を作るか決める | 人間 |
| アーキテクチャの最終判断 | 人間 |
| 既存コードの読み解き・検索 | Claude Code |
| ボイラープレートの実装 | Claude Code |
| Win32 API の細かい仕様調査 | Claude Code |
| テストケースの設計 | 人間 ＋ Claude Code |
| デバッグ仮説の列挙 | Claude Code |
| 実際に動作確認する | 人間 |

Claude Code は「コードを書く」より「コードを読む・調べる」作業の高速化に
最も価値があると感じた。特に、Windows 固有の API（ConPTY・IME・GDI）の
ドキュメントや使用例の収集は、人間が一人でやると時間がかかりすぎる。

## ハマりポイントと対策

### コンテキスト窓の制限

長い会話の中でファイルを何度も読み直すと、過去の情報が押し出されて
整合性が取れなくなることがある。

対策: 会話を適度に分割し、CLAUDE.md / task.md に必要な情報を外部化する。
新しい会話を始めるときはこれらのファイルを参照させるだけで状態を復元できる。

### 「動くが正しくないコード」の混入

Claude Code はテストが通るコードを素早く生成するが、
コーナーケースやパフォーマンスへの配慮が不足することがある。

対策: `cargo clippy -- -D warnings` を CI に組み込み、
Clippy が警告を出したら人間がレビューする。

### 外部クレートの最新情報

学習データの期限以降にリリースされた API については誤情報を返すことがある。

対策: クレートのバージョンは `cargo search` / `crates.io` で人間が確認する。
「最新は何か？」という質問は Claude Code ではなく、直接 crates.io で調べる。

## まとめ

AI エージェントとのペアプログラミングは「速度の向上」よりも
**「到達できる複雑さの上限の引き上げ」**に価値がある。

Windows ネイティブのターミナルマルチプレクサは、
一人で作ろうとすると Win32 API・ConPTY・VT エスケープ・CJK 文字幅・
非同期 I/O といった複数の難しい領域を同時に扱う必要がある。
Claude Code があることで、ある領域を実装している間に別の領域の知識不足を
補えるようになり、プロジェクトを最後まで完走できた。
